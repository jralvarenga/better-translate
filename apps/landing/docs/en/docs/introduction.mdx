# Introduction

Most i18n libraries treat TypeScript as an afterthought. You wire up your translations, then separately bolt on type safety through declaration files, module augmentation, and a handful of workarounds. If you've ever typed `t("account.balanc")` and only found out at runtime, you know how that story ends.

Better Translate flips that. Type safety isn't a plugin or a config flag here — it's the foundation everything is built on.

## The core idea

Configure your translations once. Get typed keys, typed interpolation params, fallback locale support, and async loading — everywhere, with zero boilerplate.

```tsx
import { configureTranslations } from "better-translate"

const en = {
  greeting: "Hello, {name}!",
  nav: { home: "Home", about: "About" },
}

const es = {
  greeting: "¡Hola, {name}!",
  nav: { home: "Inicio", about: "Acerca de" },
}

const translator = await configureTranslations({ en, es })
const t = translator.getTranslator("en")

t("greeting", { params: { name: "World" } }) // ✅ "Hello, World!"
t("nav.home")                                 // ✅ "Home"
t("nav.hoome")                                // ❌ TypeScript error — instantly
t("greeting")                                 // ❌ TypeScript error — params.name is required
```

That last two lines aren't runtime errors caught in QA. They're compile-time errors caught the moment you type them. That's the difference.

## Why not just use i18next or next-intl?

They're great libraries — widely adopted, battle-tested, and full of features. But type safety in both is opt-in and additive:

- **i18next** requires you to create a `i18next.d.ts` file, augment `CustomTypeOptions`, and register your resources manually. Miss a step and you're back to `string`.
- **next-intl** needs a `global.d.ts` with manually imported message types. It works, but it's setup you maintain separately from your actual translations.
- **Interpolation params** are rarely typed at all without extra tooling in either library.

With Better Translate, there's nothing to register or augment. Your locale objects *are* the source of truth. TypeScript infers everything from them automatically — keys, nested paths, and interpolation variables included.

## One config, your whole stack

This is the part that sets Better Translate apart from most alternatives: the core package is completely framework-agnostic. It runs in Node.js, Bun, plain TypeScript, APIs, and server environments without modification.

The React and Next.js packages aren't separate products with their own opinions — they're optimized layers on top of the same core. The same `configureTranslations` call you use in your backend is what powers your React context and your Next.js App Router routing. One configuration, one source of truth, consistent behavior everywhere.

```
better-translate          ← Your single source of truth. Works anywhere TypeScript runs.
    ↓
@better-translate/react   ← Adds provider + useTranslations() hook. Powered by core.
    ↓
@better-translate/nextjs  ← Adds locale routing + server helpers. Powered by core.
```

Scale from a simple API to a full Next.js app without ever rewriting your translation config.

## What you get out of the box

- **Typed keys** — autocomplete and compile-time errors for every key, including nested paths like `nav.home`
- **Typed interpolation** — if a message takes `{name}`, passing it is required; passing the wrong thing is an error
- **Fallback locale support** — define once, missing translations gracefully fall back
- **Async locale loading** — load only what you need, when you need it
- **Framework adapters** — React, Next.js App Router, and TanStack Router, all derived from the same core
- **VS Code extension** — navigate translation keys inline, without leaving your editor
