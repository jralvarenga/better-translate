# RTL

`better-translate` has built-in RTL support.
You declare which locales are right-to-left in config, then query direction at runtime
using `getDirection` or `isRtl`.

## When to use it

Use RTL support when your app needs to support languages like Arabic, Hebrew, Persian, or Urdu.
The direction helpers give you a typed way to set the `dir` attribute on HTML elements
without managing a separate lookup table.

## Declaring directions

Add a `directions` key to your `configureTranslations` call. Any locale you omit defaults to `"ltr"`.

```tsx showLineNumbers
import { configureTranslations } from "better-translate"

const translator = await configureTranslations({
  availableLocales: ["en", "ar"] as const,
  defaultLocale: "en",
  messages: { en, ar },
  directions: {
    ar: "rtl",
  },
})
```

## Querying direction

Call `getDirection` to get `"ltr"` or `"rtl"` for any locale.
Call `isRtl` to get a boolean shorthand.

```tsx showLineNumbers
translator.getDirection({ locale: "ar" }) // "rtl"
translator.getDirection({ locale: "en" }) // "ltr"

translator.isRtl({ locale: "ar" }) // true
translator.isRtl({ locale: "en" }) // false
```

## Per-call override

Pass `config.rtl` to override the configured direction for a single call.
This does not affect message lookup — it only changes what direction is resolved.

```tsx showLineNumbers
// Force LTR even for an RTL locale
translator.t("greeting", {
  locale: "ar",
  config: { rtl: false },
})

// Force RTL even for an LTR locale
translator.t("greeting", {
  locale: "en",
  config: { rtl: true },
})
```

## Applying to HTML

Use `getDirection` when rendering the `dir` attribute on root elements.

```tsx showLineNumbers
// On the html element (e.g. in a Next.js layout)
<html lang={locale} dir={translator.getDirection({ locale })}>

// On a specific container
<div dir={translator.getDirection({ locale })}>
  {t("greeting")}
</div>
```
