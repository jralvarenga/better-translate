# المقدمة

معظم مكتبات i18n تعامل TypeScript كفكرة لاحقة. تقوم بإعداد ترجماتك، ثم تضيف أمان النوع بشكل منفصل من خلال ملفات التعريف، وزيادة الوحدات، وعدة حلول بديلة. إذا كنت قد كتبت `t("account.balanc")` يوماً ولم تكتشف الخطأ إلا في وقت التشغيل، فأنت تعرف كيف تنتهي هذه القصة.

Better Translate تقلب هذا المفهوم رأساً على عقب. أمان النوع ليس مكوناً إضافياً أو علامة ضبط هنا ,  إنه الأساس الذي يُبنى عليه كل شيء.

## الفكرة الأساسية

اضبط ترجماتك مرة واحدة. احصل على مفاتيح مكتوبة النوع، ومعاملات الاستيفاء المكتوبة، ودعم اللغة الاحتياطية، والتحميل غير المتزامن ,  في كل مكان، بدون كود نمطي.

```tsx
import { configureTranslations } from "better-translate"

const en = {
  greeting: "Hello, {name}!",
  nav: { home: "Home", about: "About" },
}

const ar = {
  greeting: "مرحباً، {name}!",
  nav: { home: "الرئيسية", about: "حول" },
}

const translator = await configureTranslations({ en, ar })
const t = translator.getTranslator("ar")

t("greeting", { params: { name: "عالم" } }) // ✅ "مرحباً، عالم!"
t("nav.home")                                // ✅ "الرئيسية"
t("nav.hoome")                               // ❌ خطأ TypeScript ,  فوراً
t("greeting")                                // ❌ خطأ TypeScript ,  params.name مطلوب
```

آخر سطرين ليسا أخطاء وقت تشغيل تُكتشف في مرحلة ضمان الجودة. إنهما أخطاء وقت الترجمة تُكتشف في اللحظة التي تكتبهما فيها. هذا هو الفرق.

## لماذا لا تستخدم i18next أو next-intl؟

إنهما مكتبتان رائعتان ,  مستخدمتان على نطاق واسع، واختُبرتا في المعارك، ومليئتان بالميزات. لكن أمان النوع في كليهما اختياري وإضافي:

- **i18next** يتطلب منك إنشاء ملف `i18next.d.ts`، وزيادة `CustomTypeOptions`، وتسجيل مواردك يدوياً. تخطَّ خطوة واحدة وستعود إلى `string`.
- **next-intl** يحتاج إلى `global.d.ts` مع أنواع الرسائل المستوردة يدوياً. يعمل، لكنه إعداد تحافظ عليه بشكل منفصل عن ترجماتك الفعلية.
- **معاملات الاستيفاء** نادراً ما تكون مكتوبة النوع على الإطلاق بدون أدوات إضافية في أي من المكتبتين.

مع Better Translate، لا يوجد شيء للتسجيل أو الزيادة. كائنات اللغة الخاصة بك *هي* مصدر الحقيقة. TypeScript يستنتج كل شيء منها تلقائياً ,  المفاتيح والمسارات المتداخلة ومتغيرات الاستيفاء تشمل.

## إعداد واحد، طوال المشروع

هذا هو الجزء الذي يميز Better Translate عن معظم البدائل: الحزمة الأساسية مستقلة تماماً عن الإطار. تعمل في Node.js وBun وTypeScript العادي وAPIs وبيئات الخادم بدون تعديل.

حزم React وNext.js ليست منتجات منفصلة بآرائها الخاصة ,  إنها طبقات محسّنة فوق نفس النواة. نفس استدعاء `configureTranslations` الذي تستخدمه في الخلفية هو ما يُشغّل سياق React الخاص بك وتوجيه Next.js App Router. إعداد واحد، مصدر حقيقة واحد، سلوك متسق في كل مكان.

```
better-translate          ← مصدر الحقيقة الواحد. يعمل في أي مكان يعمل فيه TypeScript.
    ↓
@better-translate/react   ← يضيف الـ provider + useTranslations(). مدعوم بالنواة.
    ↓
@better-translate/nextjs  ← يضيف توجيه اللغة + مساعدات الخادم. مدعوم بالنواة.
```

انتقل من API بسيط إلى تطبيق Next.js كامل دون إعادة كتابة إعداد الترجمة مطلقاً.

## ما تحصل عليه جاهزاً

- **مفاتيح مكتوبة النوع** ,  إكمال تلقائي وأخطاء وقت الترجمة لكل مفتاح، بما في ذلك المسارات المتداخلة مثل `nav.home`
- **استيفاء مكتوب النوع** ,  إذا كانت رسالة تأخذ `{name}`، فتمريرها مطلوب؛ تمرير الشيء الخاطئ هو خطأ
- **دعم اللغة الاحتياطية** ,  حدد مرة واحدة، والترجمات المفقودة تعود تلقائياً إلى اللغة الاحتياطية
- **تحميل اللغة غير المتزامن** ,  حمّل فقط ما تحتاجه، عندما تحتاجه
- **محولات الإطار** ,  React وNext.js App Router وTanStack Router، جميعها مشتقة من نفس النواة
- **امتداد VS Code** ,  تنقل بين مفاتيح الترجمة مباشرة، بدون مغادرة المحرر
