# Introducción

La mayoría de las librerías de i18n tratan a TypeScript como algo secundario. Conectas tus traducciones y luego, aparte, intentas añadir type safety con archivos de declaración, *module augmentation* y varios *workarounds*. Si alguna vez escribiste `t("account.balanc")` y te enteraste recién en runtime, ya sabes cómo termina esa historia.

Better Translate le da la vuelta a eso. Aquí el type safety no es un plugin ni un flag de configuración: es la base sobre la que está construido todo.

## La idea principal

Configura tus traducciones una sola vez. Obtén claves tipadas, parámetros de interpolación tipados, soporte de locale de fallback y carga asíncrona… en todas partes, sin *boilerplate*.

```tsx
import { configureTranslations } from "better-translate"

const en = {
  greeting: "Hello, {name}!",
  nav: { home: "Home", about: "About" },
}

const es = {
  greeting: "¡Hola, {name}!",
  nav: { home: "Inicio", about: "Acerca de" },
}

const translator = await configureTranslations({ en, es })
const t = translator.getTranslator("en")

t("greeting", { params: { name: "World" } }) // ✅ "Hello, World!"
t("nav.home")                                 // ✅ "Home"
t("nav.hoome")                                // ❌ Error de TypeScript — al instante
t("greeting")                                 // ❌ Error de TypeScript — params.name es obligatorio
```

Esas dos últimas líneas no son errores en tiempo de ejecución que se descubren en QA. Son errores de compilación que se detectan en el momento en que las escribes. Esa es la diferencia.

## ¿Por qué no simplemente usar i18next o next-intl?

Son excelentes librerías — muy adoptadas, probadas en batalla y llenas de funcionalidades. Pero el type safety en ambas es opcional y aditivo:

- **i18next** requiere que crees un archivo `i18next.d.ts`, que hagas *augment* de `CustomTypeOptions` y registres tus recursos manualmente. Si te saltas un paso, vuelves a `string`.
- **next-intl** necesita un `global.d.ts` con tipos de mensajes importados manualmente. Funciona, pero es configuración que mantienes separada de tus traducciones reales.
- **Los parámetros de interpolación** casi nunca están tipados sin tooling extra en ninguna de las dos.

Con Better Translate no hay nada que registrar ni aumentar. Tus objetos de locales *son* la fuente de la verdad. TypeScript infiere todo a partir de ellos automáticamente: claves, rutas anidadas y variables de interpolación incluidas.

## Una sola configuración para todo tu *stack*

Esto es lo que diferencia a Better Translate de la mayoría de alternativas: el paquete core es completamente agnóstico al framework. Funciona en Node.js, Bun, TypeScript puro, APIs y entornos de servidor sin modificaciones.

Los paquetes de React y Next.js no son productos separados con su propia opinión: son capas optimizadas encima del mismo core. La misma llamada a `configureTranslations` que usas en tu backend es la que alimenta tu contexto de React y el enrutado de App Router de Next.js. Una sola configuración, una sola fuente de verdad, comportamiento consistente en todas partes.

```
better-translate          ← Tu única fuente de verdad. Funciona donde corra TypeScript.
    ↓
@better-translate/react   ← Añade provider + hook useTranslations(). Impulsado por el core.
    ↓
@better-translate/nextjs  ← Añade routing de locales + helpers de servidor. Impulsado por el core.
```

Escala desde una API sencilla hasta una app completa en Next.js sin reescribir nunca tu configuración de traducciones.

## Qué obtienes desde el primer momento

- **Claves tipadas** — autocompletado y errores de compilación para cada clave, incluyendo rutas anidadas como `nav.home`
- **Interpolación tipada** — si un mensaje recibe `{name}`, pasarla es obligatorio; pasar algo incorrecto es un error
- **Soporte de locale de fallback** — lo defines una vez y las traducciones faltantes hacen fallback de forma controlada
- **Carga asíncrona de locales** — carga solo lo que necesitas, cuando lo necesitas
- **Adaptadores de framework** — React, Next.js App Router y TanStack Router, todos derivados del mismo core
- **Extensión para VS Code** — navega claves de traducción en línea sin salir del editor

## Mapa de paquetes

| Paquete | Qué hace |
|---|---|
| `better-translate` | Motor core. Agnóstico al framework. La base de todo lo demás. |
| `@better-translate/react` | Provider de React y hook `useTranslations()` para cambio de locale en el cliente. |
| `@better-translate/nextjs` | Helpers para App Router: routing de locales, helpers de servidor, navegación. |
| `@better-translate/tanstack-router` | Adaptador para TanStack Router, compatible con TanStack Start. |
| `@better-translate/vscode` | Extensión de VS Code para navegar claves de traducción y ver vistas previas en línea. |
